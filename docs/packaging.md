# Packaging and WAX206 Test Build

This project is packaged as a normal OpenWrt source package. The package source is the `openwrt-squid-profile-ui/` directory. That directory is the only source of truth for package files.

## Package Layout

```text
openwrt-squid-profile-ui/
├── Makefile
├── LICENSE
└── files/
    ├── etc/
    │   ├── config/squid_profiles
    │   ├── init.d/squid-profiles
    │   └── uci-defaults/90_squid_profiles
    ├── usr/
    │   ├── lib/lua/luci/controller/squid_profiles.lua
    │   ├── libexec/squid-profiles
    │   └── share/
    │       ├── luci/menu.d/luci-app-squid-profiles.json
    │       └── rpcd/acl.d/luci-app-squid-profiles.json
    └── www/luci-static/resources/view/squid-profiles/
        ├── main.js
        ├── networks.js
        └── profiles.js
```

There is no `src/` directory because this package does not compile target binaries. It installs LuCI JavaScript, a small Lua controller, shell helpers, a UCI config file, an init script and ACL/menu metadata.

## Package Makefile Model

The package follows the OpenWrt source package model:

- `PKG_NAME:=luci-app-squid-profiles`
- `PKG_VERSION:=0.1.0`
- `PKG_RELEASE:=2`
- `PKG_LICENSE:=BSD-2-Clause`
- `PKG_LICENSE_FILES:=LICENSE`
- `PKGARCH:=all`
- `Package/luci-app-squid-profiles/install` copies files into the package root under `$(1)`
- `Package/luci-app-squid-profiles/conffiles` marks `/etc/config/squid_profiles` as configuration

Runtime dependencies are declared in `DEPENDS`:

```make
+luci-base +luci-compat +luci-lua-runtime +rpcd +uci +squid
```

The package is architecture-independent because it ships scripts and LuCI resources only. Build it with the SDK matching the target router anyway, because the router will install dependencies from that same OpenWrt release and target feed.

## OpenWrt 25 SDK Docker Build

This repository targets OpenWrt 25 and APK packages. The GitHub Actions workflow uses `openwrt/gh-action-sdk`, which wraps the official OpenWrt SDK containers.

The SDK container tags follow this pattern:

```text
openwrt/sdk:<target>-<subtarget>-<branch-or-version>
```

For Netgear WAX206 development, use the Mediatek MT7622 SDK from the OpenWrt main branch:

```sh
mkdir -p bin
chmod 0777 bin
docker run --rm \
  -v "$(pwd)":/work:ro \
  -v "$(pwd)/bin":/builder/bin \
  -it openwrt/sdk:mediatek-mt7622-main
```

Inside the container:

```sh
[ ! -d ./scripts ] && ./setup.sh
mkdir -p package/feeds/custom
cp -a /work/openwrt-squid-profile-ui package/feeds/custom/luci-app-squid-profiles
./scripts/feeds update packages luci
./scripts/feeds install luci-app-squid-profiles
make defconfig
make package/luci-app-squid-profiles/check V=s
make package/luci-app-squid-profiles/clean V=s
make package/luci-app-squid-profiles/compile V=s -j"$(nproc)"
make package/index V=s
find bin/packages -name 'luci-app-squid-profiles_*.apk' -print
```

The host receives package output in `./bin` because it is mounted to `/builder/bin`.

## Simple local deployment from GitHub Actions output

The GitHub Actions workflow publishes two artifact sets:

- the package archive containing `luci-app-squid-profiles_*.apk`
- the repository metadata generated by `make package/index`

For direct router installation, you only need the package archive.

Download the workflow artifact, then copy the APK to the router over SSH:

```sh
scp luci-app-squid-profiles_*.apk root@192.168.1.1:/tmp/
ssh root@192.168.1.1
apk update
apk add --allow-untrusted /tmp/luci-app-squid-profiles_*.apk
```

Then validate the generated Squid configuration:

```sh
/usr/libexec/squid-profiles validate
squid -k parse
```

## Downloading the latest release with `wget`

The workflow also publishes release assets for tag builds. To fetch the latest generated APK with `wget`, ask the GitHub API for the newest release asset URL and download it:

```sh
APK_URL="$(wget -qO- "https://api.github.com/repos/${GITHUB_REPOSITORY}/releases/latest" | sed -n 's/.*"browser_download_url": *"\([^"]*luci-app-squid-profiles_[^"]*\.apk\)".*/\1/p' | head -n1)"
wget -O /tmp/luci-app-squid-profiles.apk "$APK_URL"
```

If you publish the repository index as a release asset too, use the same pattern for `packages.adb` or `APKINDEX.tar.gz`.

For repository-based installs, publish both the APK package and the repository metadata generated by `make package/index`, then point the router at that repository using the APK repository configuration supported by the installed OpenWrt 25 build.

## Package Source and Feed Publication

The repository intentionally does not keep a second package tree under `packages/`. The CI workflow and local Docker instructions copy the single source directory into the SDK:

```sh
mkdir -p package/feeds/custom
cp -a /work/openwrt-squid-profile-ui package/feeds/custom/luci-app-squid-profiles
```

If you want to publish this as a conventional OpenWrt feed, publish a feed repository where the package directory is available at the feed root as `luci-app-squid-profiles/`, or mirror `openwrt-squid-profile-ui/` into that name during feed generation. Avoid keeping two editable package copies in this repository.

## GitHub Actions

The workflow `.github/workflows/openwrt-package.yml` builds the package with `openwrt/gh-action-sdk` and uploads:

- generated APK package files: `*.apk`
- repository metadata generated by `make package/index`

It targets the OpenWrt 25 `aarch64_cortex-a53` SDK class, which fits the WAX206 family.

## APK Repository Metadata and Signing

OpenWrt 25 uses APK packages. The OpenWrt SDK generates APK package files and repository metadata when `make package/index` runs. Depending on the SDK snapshot, repository metadata may include files such as:

```text
packages.adb
packages.adb.sig
APKINDEX.tar.gz
APKINDEX.tar.gz.sig
```

Publish the generated APK files and repository metadata together. OpenWrt signs APK repository indexes, not this LuCI package source tree. OpenWrt public-key documentation describes APK keys as EC keys using the NIST P-256 curve for signing `packages.adb` indexes.

When the selected SDK/buildroot is configured with signing keys, `make package/index` should produce signed metadata. If you need to sign or re-sign an index manually in a controlled release job, use the APK tooling from the matching SDK/buildroot, for example:

```sh
apk adbsign --allow-untrusted --sign-key /path/to/apk-private.pem packages.adb
```

Then publish the signed index and install the matching public key on the router according to the OpenWrt 25 APK keyring/repository process used by that firmware build.

## Publication Layout

Use one directory per OpenWrt release and target:

```text
public/
└── openwrt/
    └── 25-main/
        └── mediatek-mt7622/
            ├── luci-app-squid-profiles_*.apk
            └── packages.adb
```

Keep feeds separate per OpenWrt branch/version and target. Dependency resolution must match the router firmware and APK repository set.

## Manual SSH Install Sequence

Use this when you only want to install the workflow result on one router:

1. Download the latest release asset with the `wget` command above, or download the `*.apk` artifact from GitHub Actions.
2. Copy it to the router:

```sh
scp luci-app-squid-profiles_*.apk root@192.168.1.1:/tmp/
```

3. Install it:

```sh
ssh root@192.168.1.1
apk add --allow-untrusted /tmp/luci-app-squid-profiles_*.apk
```

4. Validate the generated Squid config:

```sh
/usr/libexec/squid-profiles validate
squid -k parse
```

5. Apply only after validation succeeds:

```sh
/usr/libexec/squid-profiles apply
```

## Netgear WAX206 Build

The Netgear WAX206 is built by OpenWrt under the `mediatek/mt7622` target. For OpenWrt 25 development, use:

```text
openwrt/sdk:mediatek-mt7622-main
```

Reference URLs:

- `https://github.com/openwrt/docker`
- `https://hub.docker.com/r/openwrt/sdk`
- `https://openwrt.org/docs/techref/targets/mediatek`

Build:

```sh
docker run --rm -v "$(pwd)":/work:ro -v "$(pwd)/bin":/builder/bin -it openwrt/sdk:mediatek-mt7622-main
[ ! -d ./scripts ] && ./setup.sh
mkdir -p package/feeds/custom
cp -a /work/openwrt-squid-profile-ui package/feeds/custom/luci-app-squid-profiles
./scripts/feeds update packages luci
./scripts/feeds install luci-app-squid-profiles
make defconfig
make package/luci-app-squid-profiles/check V=s
make package/luci-app-squid-profiles/compile V=s
make package/index V=s
find bin/packages -name 'luci-app-squid-profiles_*.apk' -print
```

Transfer the APK to the WAX206:

```sh
scp bin/packages/*/*/luci-app-squid-profiles_*.apk root@192.168.1.1:/tmp/
ssh root@192.168.1.1
apk update
apk add --allow-untrusted /tmp/luci-app-squid-profiles_*.apk
/usr/libexec/squid-profiles init
/usr/libexec/squid-profiles validate
```

Open LuCI:

```text
Services -> Squid Profiles
```

Expected tabs:

- `Profiles`
- `Devices`
- `LAN/VLAN Mapping`

## Router-Side Validation

After installation or changes:

```sh
uci show squid_profiles
/usr/libexec/squid-profiles validate
/usr/libexec/squid-profiles apply
squid -k parse
logread
```

The plugin must not reload Squid unless validation succeeds.

## Common Build Problems

- Wrong SDK image: use the `openwrt/sdk` tag matching the target and OpenWrt 25 branch used by the router firmware.
- Missing dependencies at install: run `apk update` and ensure the router repositories match the firmware branch.
- Local changes not included: the Docker workflow copies `openwrt-squid-profile-ui/` into the SDK container; rerun the container build after edits.
- Stale package metadata: rerun `./scripts/feeds update packages luci` and `./scripts/feeds install luci-app-squid-profiles`.
- Wrong package manager: this project now targets OpenWrt 25/APK only.
